This document provides the **technical specification** for the tagging and query system used in Icarus Calculator. External engineers should use this as a reference to implement a compatible tag query engine for their applications.

## 1. Naming Conventions

Tags follow a specific naming protocol to indicate their origin and purpose.

### IC.\* Prefix (Calculated/Inherited)

Tags prefixed with `IC.` are generated by the `DataGenerator` during the extraction process. These are usually derived from metadata, implied by item properties (like being a food ingredient), or propagated through the recipe graph.

### \*.IC Suffix (Manual Corrections)

Tags suffixed with `.IC` (e.g., `Item.Plant.Coconut.IC`) indicate manual corrections or simplifications. These are applied when the raw game data is missing a logical tag or uses an inconsistent naming convention that we've standardized for the calculator.

### Traits.\* (Legacy)

> [!WARNING]
> `Traits.*` tags are legacy markers from earlier versions of the game's data structure. They are strictly deprecated and slated for removal in a future update.

---

### Implementation Requirements

Client-side engines **must** implement hierarchical matching to correctly interpret the dataset:

- **Rule**: When querying for a tag, the engine **must** match any item that has the **specified tag** OR any **child tag** (dot-notation).
- **Case Insensitivity**: Tags in the Icarus ecosystem are **case-insensitive**. Engines should normalize all data and queries to a consistent case (typically lowercase) before evaluation.

**Matching Example:**

- Querying for `Item.Plant` matches:
  - `Item.Plant`
  - `Item.Plant.Fruit`
  - `Item.Plant.Vegetable.IC`

This allows for broad filtering (e.g., "all plants") without needing to list every specific sub-type.

## 3. Tag Query Engine

The `TagQueryEngine` evaluates complex boolean expressions against an item's tag set.

### Operators

Query operators are case-insensitive and can be nested.

| Operator       | Description | Matches If...                                                     |
| :------------- | :---------- | :---------------------------------------------------------------- |
| `ANY(t1, t2)`  | OR Logic    | Item has **at least one** of the listed tags (or their children). |
| `ALL(t1, t2)`  | AND Logic   | Item has **all** of the listed tags (or their children).          |
| `NONE(t1, t2)` | NOT Logic   | Item has **none** of the listed tags (or their children).         |

### Recursion Safety

Queries can be deeply nested (e.g., `ANY(ALL(A, B), NONE(C))`). To prevent infinite loops or stack overflows from malformed data, the engine enforces a strict **limit of 20 levels of recursion**.

## 4. IC-Generated Categories

The following `IC.*` tags are automatically managed by the generator logic.

### IC.Category.\* (Logical Grouping)

Used to group items for the UI "Category" dropdowns.

- `IC.Category.AnimalParts`: Derived from hunting/skinning (Bone, Leather, Fur, etc.).
- `IC.Category.Drink`: Items that typically replenish `BaseWaterRecovery`.
- `IC.Category.Food`: Items with nutritional stats or food-based modifiers.
- `IC.Category.Misc`: A catch-all for items that don't fit standard gameplay roles.
- `IC.Category.Plant`: Foraged or cultivated vegetation.
- `IC.Category.Resources`: Raw materials used across multiple tiers (Ore, Wood, Stone).

### IC.Material.\* (State Markers)

- `IC.Material.Ingot`: Refined metals.
- `IC.Material.Raw`: Unprocessed ores or materials.

### IC.Marker.\* (Processing Logic)

- `IC.Marker.Container`: Items that act as vessels for other resources (primarily fluids/oxygen).
- `IC.Marker.Food.Ingredient`: Items that are not food themselves but are required inputs for a recipe that outputs `IC.Category.Food`.

### IC.Required.\* (Access Control)

- `IC.Required.FeatureLevel.*`: Indicates the DLC or Expansion required (e.g., `Styx`, `NewFrontiers`).
- `IC.Required.Workshop.Meta.*`: Indicates the Workshop currency required (e.g., `Ren`, `Exotics`).

## 5. Special Handling Logic

### The Fish Consolidation

To simplify the complex game fish data, the calculator consolidates all raw fish into `IC.Fish.Raw`. Manual `.IC` suffixes are applied to disambiguate Freshwater/Saltwater/Unique variants where the game data is incomplete.

### The Fluid Measurement Standards

Fluids (Water, Fuel, etc.) represent a unique data type in Icarus Calculator.

- **Internal Storage**: All fluid amounts are processed and stored in **Milliliters (ML)** to maintain high precision across recipe requirements.
- **Display Mapping**: Items tagged with `IC.Fluid.*` are automatically assigned a `display_operations` rule: `{"operation": "division", "value": 1000}`. This ensures the UI displays values in **Liters (L)**, which is more readable for players.
